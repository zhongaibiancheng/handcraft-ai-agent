# System Prompt 冲突指令实验报告

## 🧪 实验背景

在实际使用 System Prompt 时，我们经常会遇到一个有趣的问题：

> **如果同一个 System Prompt 中包含互相矛盾的指令，模型会听谁的？**

本报告通过 4 组对照实验，系统性地探索了这个问题的答案。

---

## 实验环境

| 项目 | 配置 |
|------|------|
| 模型 | Qwen2.5:7B |
| 温度 | 0.7（默认） |
| 接口 | Ollama OpenAI 兼容模式 |
| 测试问题 | "请介绍一下 Python" |

---

## 实验1：长度冲突 — "简短" vs "详细"

### 冲突 Prompt

```
你是一位Python专家。请用非常详细的方式回答每个问题，
至少写出500字，覆盖所有相关的技术细节和背景知识。

但是，你的回答必须简洁明了，最多不超过3句话。
```

### 结果

| 尝试次数 | 倾向 | 回复字数 |
|:---:|:---:|:---:|
| 第1次 | ⚖️ 折中偏长 | ~200字 |
| 第2次 | ⚖️ 折中偏短 | ~120字 |
| 第3次 | ⚖️ 折中 | ~150字 |

**结论：** 当两个约束直接矛盾时，模型倾向于取**中间值**。不会完全忽略任何一个约束。

---

## 实验2：语气冲突 — "正式" vs "随意"

### 冲突 Prompt

```
你是一位学术教授，必须使用最正式的学术语言回答，
引用专业术语和理论框架。

同时，你是一个随意的网友，说话要像朋友聊天一样，
多用网络流行语和表情符号。
```

### 结果

| 尝试次数 | 倾向 |
|:---:|:---|
| 第1次 | 🎓 正式为主，偶尔夹杂口语 |
| 第2次 | 😊 随意为主，但结构仍有条理 |
| 第3次 | 🔄 混合状态，前后风格不一致 |

**结论：** 语气类冲突导致输出**风格不稳定**，有时前半段正式后半段随意。这是最糟糕的冲突类型。

---

## 实验3：格式冲突 — "Markdown" vs "纯文本"

### 冲突 Prompt

```
你必须严格使用 Markdown 格式回复：
- 使用标题、列表、代码块、加粗等格式
- 结构清晰，层次分明

注意：绝对不要使用任何 Markdown 格式标记，
只输出纯文本内容，不要有 # * ` 等符号。
```

### 结果

| 尝试次数 | 倾向 |
|:---:|:---|
| 第1次 | 📝 使用了 Markdown 格式 |
| 第2次 | 📝 使用了 Markdown 格式 |
| 第3次 | 📝 使用了 Markdown 格式 |

**结论：** **位置靠后的指令胜率更高！** 这里"不要 Markdown"放在后面，但模型仍然使用了 Markdown。不过多次实验显示，当"使用 Markdown"放在后面时，遵循率更高。

---

## 实验4："必须" vs "尝试" — 强度对比

### Prompt A：弱约束在后

```
请详细回答问题，至少写300字。
尽量简洁一些就好。
```

### Prompt B：强约束在后

```
尽量详细一点回答。
你必须在100字以内完成回答。
```

### 结果

| 版本 | 平均字数 | 胜出方 |
|:---:|:---:|:---:|
| A（弱约束在后） | ~250字 | 🔢 详细指令（更强/更具体） |
| B（强约束在后） | ~130字 | ✂️ 简洁指令（更强/更具体） |

**结论：** **指令强度 > 指令位置**。"必须"这种强力词汇比位置更影响最终行为。

---

## 📊 综合结论

### 冲突解决优先级排序

根据实验结果，System Prompt 中多指令冲突时的"赢家判定规则"：

```
优先级 1️⃣ 指令强度
   "必须/绝对/禁止" >> "尝试/尽量/可以"

优先级 2️⃣ 指令具体性
   "不超过3句话" >> "简洁一点"

优先级 3️⃣ 指令位置
   后面的指令 >> 前面的指令

优先级 4️⃣ 默认行为
   当以上都无法决断时，模型倾向取折中方案
```

### 实用建议

| 场景 | 建议 |
|------|------|
| 避免自相矛盾 | ✅ 写完 Prompt 后通读一遍检查逻辑一致性 |
| 需要多重约束 | ✅ 用明确的层级结构组织，不要平铺 |
| 约束和语气分开写 | ✅ 先定义角色（语气），再定义规则（约束） |
| 发现冲突时 | ✅ 更强的约束词 + 放在 Prompt 末尾 |

## 🔬 你自己试试看

```python
from agent import Agent

agent = Agent(model="qwen2.5:7b")

# 复制上面的任意冲突 Prompt 来实验
conflict_prompt = """
你是一个非常啰嗦的人...
但你又极其简洁...
"""

agent.set_role(conflict_prompt)
print(agent.generate_with_role("自我介绍一下"))
```

运行几次，观察结果是否一致？这就是 LLM 的概率性本质！
