# System Prompt 最佳实践指南

## 📐 写好 System Prompt 的 7 条黄金法则

---

### 法则1：角色 + 约束 + 格式，三件套不能少

一个高质量的 System Prompt 应该包含三个层次：

```
【第一层：你是谁】
你是一位资深Python开发工程师...

【第二层：你的规则】
- 回答必须包含可运行的代码示例
- 每段代码必须加中文注释
- 遇到错误要说明原因和修复方法

【第三层：输出格式】
- 先给直接答案
- 再给代码示例
- 最后给注意事项
- 总字数不超过200字
```

**为什么三件套都要有？**
- 只有角色 → 模型知道"是谁"，但不知道"怎么做"
- 只有约束 → 模型知道限制，但没有方向感
- 只有格式 → 输出整齐但内容可能空洞

---

### 法则2：用具体替代模糊

这是新手最容易犯的错误：

| ❌ 模糊表达 | ✅ 具体表达 |
|-----------|-----------|
| 专业一点 | 使用学术风格的语言，每段引用至少一个技术术语 |
| 简短回答 | 用2-3句话回答，不要超过100字 |
| 给点例子 | 每个概念给出1个实际应用场景的示例代码 |
| 详细解释 | 从原理、实现、优缺点三个角度分别阐述 |

**口诀：能数字化的就数字化，能举例子的就举例子。**

---

### 法则3：强语气词 > 弱语气词

指令的"力度"直接影响遵循率：

| 力度 | 词汇 | 适用场景 |
|:---:|------|---------|
| 🔴 最强 | **绝对禁止 / 必须 / 务必 / 严格** | 格式约束、安全规则、不可妥协的要求 |
| 🟡 中等 | **应该 / 需要 / 确保** | 一般行为规范 |
| 🟢 较弱 | **可以 / 尝试 / 尽量** | 建议性引导（模型可能不遵守） |

> **实战技巧：** 关键约束用最强语气，辅助建议可以用弱语气。不要全部都用"必须"，否则会互相冲突。

---

### 法则4：结构化格式 > 大段文字

模型对结构化的文本理解更好：

**❌ 大段式（容易丢失信息）：**

```
你是一个有帮助的技术助手。你需要用简单的方式解释概念并且使用代码示例。
每次回答不要太长也不要太短，最好控制在合适的范围内。
如果有需要注意的地方一定要指出来。
```

**✅ 结构化（清晰易遵循）：**

```
你是一位技术导师。

【回答要求】
1. 用通俗语言解释概念（1段）
2. 提供可运行的代码示例（1段）
3. 指出常见踩坑点（2-3条）

【格式约束】
- 每段不超过3句话
- 总字数 < 150字
```

---

### 法则5：位置很重要 — 重要约束放最后

模型对 Prompt 中**靠后的内容关注度更高**：

```
❌ 低效顺序：
[一般介绍] → [重要格式约束] → [补充说明]

✅ 高效顺序：
[一般介绍] → [补充说明] → [重要格式约束]  ← 放最后！
```

**原因：** LLM 的注意力机制对近期的 token 赋予更高权重。把最关键的规则放在 Prompt 的末尾，效果最好。

---

### 法则6：长度控制在 200 字以内

| 长度 | 问题 | 建议 |
|-----|------|------|
| < 50字 | 信息量不足 | 至少覆盖角色+核心约束 |
| **50~200字** | — | ✅ **最佳区间** |
| 200~500字 | 注意力分散 | 删减次要信息，保留核心规则 |
| > 500字 | 严重浪费上下文窗口 | 拆分成多个 Agent |

> 一条经验法则：**如果你觉得 System Prompt 写得像一篇文章了，那它就太长了。**

---

### 法则7：先测试再上线

写完 System Prompt 后，至少做这3组测试：

```python
agent = Agent(model="qwen2.5:7b")
agent.set_role(your_prompt)

# 测试1：正常问题
print(agent.generate_with_role("什么是 Python？"))

# 测试2：边缘问题（偏门问题）
print(agent.generate_with_role("Python 的 GIL 是什么？"))

# 测试3：对抗性测试（故意问无关问题）
print(agent.generate_with_role("今天天气怎么样？"))
```

**观察要点：**
- 三次回复的风格一致吗？
- 格式约束都遵守了吗？
- 对无关问题的处理符合预期吗？（比如老师角色应该说"这个问题超出我的教学范围"而不是瞎答）

## 🎯 不同场景的 System Prompt 模板

### 场景A：技术文档生成器

```
你是一位技术文档撰写专家。

【写作规范】
- 使用清晰的标题层级（H2/H3）
- 代码块必须标注语言类型
- API 说明按 参数/返回值/示例 的顺序组织
- 每个函数说明包含：功能概述 + 参数表 + 使用示例

【质量标准】
- 无错别字和语法错误
- 技术术语准确统一
- 新手也能看懂
```

### 场景B：数据分析师

```
你是一位资深数据分析师。

【工作流程】
1. 先明确分析目标和数据背景
2. 选择合适的分析方法
3. 呈现结果时包含：结论 + 数据支撑 + 可视化建议
4. 最后指出分析的局限性

【输出格式】
- 数字保留2位小数
- 百分比使用 % 符号
- 结论先行，细节在后
```

### 场景C：创意头脑风暴

```
你是一位富有创造力的产品创新顾问。

【思维原则】
- 不否定任何想法，先发散再收敛
- 每个想法从用户价值和技术可行性两个维度评价
- 鼓励跨界类比（从其他行业找灵感）
- 提供"保守方案"+"大胆方案"两种选择

【禁忌】
- 不要说"这个做不到"
- 不要用"传统做法是..."来限制思维
```

## ⚠️ 常见反模式（千万别这么写）

| 反模式 | 为什么不好 | 正确做法 |
|--------|----------|---------|
| "像XXX一样" | 模型对 XXX 的理解和你不一样 | 直接描述期望的行为特征 |
| "不要做得太差" | "差"的标准因人而异 | 明确定义"好"的标准是什么 |
| 塞入10条以上规则 | 规则越多遵循率越低 | 只保留最重要的3-5条 |
| 角色和约束矛盾 | 模型不知道听谁的 | 确保所有规则指向同一方向 |
| 完全没有格式要求 | 输出不可预测 | 至少定义长度和基本结构 |
